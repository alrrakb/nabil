## 1 — النظرة العامة للمشروع

### ما هو النظام ده؟

نظام **إدارة المراسلات الداخلية** هو منظومة رقمية متكاملة صُممت خصيصًا لمعهد دلتا العالي، هدفها إنها تحوّل كل عمليات المراسلة الداخلية — اللي كانت بتتم على ورق أو على إيميلات متناثرة — لنظام واحد منظّم، موثّق، وآمن.

النظام بيغطي كل رحلة المراسلة من أول ما بيتم إنشاؤها، مرورًا بالمتابعة والإجراءات، لحد ما بتتأرشف. كل خطوة في العملية بتتسجّل أوتوماتيكيًا، فمش هيبقى فيه أي مراسلة بتضيع أو حد بيقول "أنا ما علمتش".

### القيمة الحقيقية اللي اتسلّمتها

| قبل النظام | بعد النظام |
|:---|:---|
| مراسلات ورقية بتتضيع أو بتتأخر | كل مراسلة ليها رقم مرجعي فريد وبتتتبّع لحظة بلحظة |
| مفيش حد عارف مين عمل إيه ومتى | سجل تاريخي كامل لكل إجراء بيتم على أي مراسلة |
| صعوبة الوصول للمراسلات القديمة | أرشيف رقمي قابل للبحث في ثواني |
| مفيش فصل واضح للصلاحيات | نظام أدوار ثلاثي (مدير — مشرف — موظف) مع حماية على مستوى قاعدة البيانات |
| تقارير يدوية مرهقة | لوحة تحكم تحليلية تُنتج التقارير بضغطة زر |
| مفيش نسخ احتياطي أو أمان | بيانات محفوظة على cloud محمي بتشفير |

## 2 — البنية التحتية والتقنيات المستخدمة

### 2.1 المعمارية العامة للنظام

النظام مبني على نموذج **ثلاثي الطبقات**:

```
┌─────────────────────────────────────────┐
│        واجهة المستخدم (Frontend)        │
│     React + Vite + Tailwind CSS         │
│         يشتغل في المتصفح                │
└────────────────┬────────────────────────┘
                 │ HTTP / REST API
┌────────────────▼────────────────────────┐
│         خادم التطبيق (API Server)       │
│         Express.js + TypeScript         │
│       بيعالج الطلبات والمنطق            │
└────────────────┬────────────────────────┘
                 │ Drizzle ORM
┌────────────────▼────────────────────────┐
│         قاعدة البيانات (Database)       │
│   PostgreSQL عبر Supabase (Cloud)       │
│     بيخزن كل البيانات بأمان            │
└─────────────────────────────────────────┘
```

### 2.2 قاعدة البيانات والأمان

النظام بيستخدم **PostgreSQL** — وهي من أقوى وأموثق قواعد البيانات في العالم، مستخدمة من شركات زي Apple وInstagram وSpotify.

البيانات بتتخزن على **Supabase** — وهي منصة cloud متخصصة في PostgreSQL بتوفر:

- **تشفير البيانات** أثناء النقل وأثناء التخزين
- **نسخ احتياطي تلقائي** يومي
- **توافر عالي (High Availability)** بنسبة 99.9%
- **Row Level Security (RLS)** — وده الميزة الأهم

#### شرح Row Level Security بلغة بزنس

فكّر في RLS كأنها حارس أمن ذكي جوه قاعدة البيانات نفسها مش بس في التطبيق. يعني حتى لو حصل اختراق للتطبيق أو حد وصل للـ API بطريقة غلط، مش هيقدر يشوف أي بيانات مش من حقه. الحماية موجودة على أعمق مستوى ممكن.

مثلًا: الموظف العادي ممنوع يشوف بيانات الموظفين التانيين أو الأرشيف — حتى لو حاول يطلبها مباشرةً من قاعدة البيانات، هيلاقي إن القاعدة نفسها ترفض الطلب.

### 2.5 هيكل قاعدة البيانات

النظام بيستخدم **5 جداول** مترابطة:

```
departments (الأقسام)
│
├── employees (الموظفون)
│   └── FK: department_id
│
└── correspondences (المراسلات)
    ├── FK: sender_id → employees
    ├── FK: receiver_id → employees
    ├── FK: department_id → departments
    │
    ├── correspondence_history (سجل الإجراءات)
    │   └── FK: correspondence_id (CASCADE DELETE)
    │
    └── archive (الأرشيف)
        └── FK: correspondence_id (CASCADE DELETE)
```

| الجدول | عدد الحقول | الوظيفة |
|:---|:---:|:---|
| `departments` | 4 | الأقسام الإدارية |
| `employees` | 8 | بيانات وصلاحيات كل موظف |
| `correspondences` | 12 | المراسلات بكل تفاصيلها |
| `correspondence_history` | 6 | سجل كل إجراء على كل مراسلة |
| `archive` | 5 | سجل الأرشفة |

## 3 — دليل النظام التفصيلي

### صفحة تسجيل الدخول `/login`

| | |
|:---|:---|
| **من يدخلها** | كل المستخدمين — صفحة عامة |
| **الهدف** | المصادقة الآمنة للدخول للنظام |

**كيف بتشتغل:**
المستخدم بيدخل إيميله وكلمة المرور، النظام بيتحقق منهم عبر Supabase Authentication. في حالة نجاح الدخول، النظام بيجيب بيانات الموظف من قاعدة البيانات ويحدد دوره (مدير/مشرف/موظف) قبل ما يوجّهه للوحة التحكم.

**مميزات تقنية:**
- رسائل خطأ واضحة بالعربي في كل الحالات (كلمة مرور غلط — إيميل مش متأكد — إلخ)
- مؤشر loading أثناء التحقق
- حماية من الدخول للصفحة لو المستخدم متسجل دخول مسبقًا (بيتحوّل للـ Dashboard أوتوماتيكيًا)

---

### لوحة التحكم الرئيسية `/`

| | |
|:---|:---|
| **من يدخلها** | جميع المستخدمين المسجلين |
| **الهدف** | نظرة شاملة ولحظية على حالة المراسلات والنظام |

**كيف بتشتغل:**
أول ما المستخدم بيدخل النظام، بيشوف لوحة تحكم فيها 8 بطاقات إحصائية و2 رسم بياني وجدول آخر المراسلات — كلها بتتحمّل بالتوازي.

**البطاقات الإحصائية الـ 8:**

| البطاقة | البيانات اللي بتعرضها |
|:---|:---|
| إجمالي المراسلات | العدد الكلي لكل المراسلات |
| قيد الانتظار | المراسلات اللي لسه متعامل معهاش |
| قيد الإجراء | المراسلات اللي شغّالة حاليًا |
| مكتملة | المراسلات اللي خلصت |
| مؤرشفة | المراسلات اللي اتأرشفت |
| عاجلة | المراسلات ذات الأولوية العاجلة |
| الأقسام | إجمالي الأقسام في النظام |
| الموظفين | إجمالي الموظفين المسجلين |

**الرسوم البيانية:**
- **Bar Chart:** توزيع المراسلات على الأقسام — بيوضح أنهي قسم عنده أكتر حجم مراسلات
- **Donut Pie Chart:** توزيع المراسلات حسب الحالة — نظرة سريعة على bottlenecks

**جدول آخر المراسلات:**
بيعرض آخر 5 مراسلات مع لينكات مباشرة لصفحة التفاصيل.

### إضافة مراسلة جديدة `/add`

| | |
|:---|:---|
| **من يدخلها** | جميع المستخدمين المسجلين |
| **الهدف** | إنشاء مراسلة جديدة في النظام |

**كيف بتشتغل:**
المستخدم بيملا فورم المراسلة، والنظام بيعمل الآتي أوتوماتيكيًا:
1. بيولّد **رقم مرجعي فريد** بصيغة `CORR-2026-XXXX`
2. بيرفع أي مرفقات على Supabase Storage
3. بيحفظ المراسلة في قاعدة البيانات
4. بيضيف أول إدخال في سجل الإجراءات تلقائيًا
5. بيوجّه المستخدم لصفحة تفاصيل المراسلة الجديدة

**حقول الفورم:**

| الحقل | إلزامي؟ | التحقق |
|:---|:---:|:---|
| العنوان | نعم | 3 حروف على الأقل |
| النوع | نعم | وارد / صادر / داخلي |
| الأولوية | نعم | منخفضة / عادية / متوسطة / عالية / عاجلة |
| المُرسِل | لا | اختيار من قائمة الموظفين |
| المستقبِل | لا | اختيار من قائمة الموظفين |
| القسم | لا | اختيار من قائمة الأقسام |
| المحتوى | لا | نص حر |
| المرفق | لا | PDF / Word / Excel / صور — حد أقصى 10MB |

### تفاصيل المراسلة `/correspondences/:id`

| | |
|:---|:---|
| **من يدخلها** | جميع المستخدمين المسجلين |
| **الهدف** | عرض كل تفاصيل مراسلة معينة وإجراء العمليات عليها |

**كيف بتشتغل:**
الصفحة دي هي "قلب" النظام — بتعرض كل شيء عن مراسلة واحدة في مكان واحد.

**قسم المعلومات الرئيسية:**
- رقم المرجع والعنوان
- النوع والتاريخ
- المُرسِل والمستقبِل
- القسم المرتبط
- محتوى / نص المراسلة
- رابط المرفق (لو موجود)
- Badge الحالة والأولوية

**الإجراءات المتاحة:**

| الإجراء | النتيجة |
|:---|:---|
| **موافقة / إتمام** | بتحوّل الحالة لـ "مكتملة" وبتسجّل الإجراء في السجل |
| **رفض** | بتحوّل الحالة لـ "مرفوضة" وبتسجّل الإجراء في السجل |
| **أرشفة** | بتنقل المراسلة للأرشيف (مع تأكيد) |

**سجل الإجراءات (Timeline):**
جنب المحتوى، فيه timeline بيعرض كل إجراء اتعمل على المراسلة ده بالترتيب الزمني — مين عمله، إيه اللي عمله، وإمتى. ده الـ **audit trail** اللي بيحمي المعهد قانونيًا ويوضح المساءلة.

### إدارة الأقسام `/departments`

| | |
|:---|:---|
| **من يدخلها** | المديرون فقط |
| **الهدف** | إدارة الهيكل الإداري للمعهد |

**كيف بتشتغل:**
بيعرض جدول بكل الأقسام المسجلة. المدير يقدر يضيف قسم جديد أو يحذف قسم موجود.

| **الهدف** | إدارة بيانات وصلاحيات كل موظف في النظام |

**كيف بتشتغل:**
بيعرض جدول بكل الموظفين مع إمكانية الإضافة والحذف.

**بيانات الموظف الجديد:**

| الحقل | إلزامي؟ | ملاحظات |
|:---|:---:|:---|
| الاسم | نعم | حرفين على الأقل |
| الإيميل | نعم | تحقق من صحة الفورمات |
| الدور | نعم | مدير / مشرف / موظف |
| القسم | لا | ممكن يكون من غير قسم |

**الأدوار وصلاحياتها في النظام:**

| الدور | الصلاحيات |
|:---|:---|
| **مدير (admin)** | وصول كامل لكل صفحات النظام بما فيها الموظفين والأقسام |
| **مشرف (supervisor)** | كل شيء عدا إدارة الموظفين والأقسام |
| **موظف (employee)** | الـ Dashboard والصندوق وإضافة مراسلة وتفاصيل المراسلة |

## 4 — مقترحات التطوير المستقبلية (المرحلة الثانية)

دي مقترحات تقنية بيزنس high-value ممكن تتضاف للنظام في المرحلة الجاية. كلها مبنية على الـ infrastructure الموجودة ومش محتاجة أي rebuild.

---

### 4.1 نظام الإشعارات الفوري (Real-time Notifications)

**الفكرة:** أي موظف بيتبعتله مراسلة أو بيتغير حالة مراسلة خاصة بيه، بيوصله إشعار فوري في المتصفح وعلى إيميله.

**التطبيق التقني:** Supabase Realtime (موجود أصلًا في الـ infrastructure) + Email via Supabase Edge Functions + Resend API.

**القيمة البيزنس:** مش هيبقى فيه تأخيرات بسبب إن حد "معرفش" إن فيه مراسلة ناطراه. وقت الاستجابة بينخفض بشكل كبير.

**التقدير:** متوسط الجهد — أسبوعين تطوير.

---

### 4.2 OCR وقراءة المرفقات الذكية

**الفكرة:** لما حد يرفع مستند PDF أو صورة كمرفق، النظام بيقرأ محتواه ويستخرج النصوص تلقائيًا ويضيفها في حقل بحث المراسلة.

**التطبيق التقني:** Google Cloud Vision API أو Tesseract.js — بيتنفّذ كـ Supabase Edge Function لما المرفق يتحمّل.

**القيمة البيزنس:** بدل ما الأرشيف يكون مجرد صور مسكانة، بيبقى أرشيف كامل البحث — تقدر تدور على كلمة معينة جوه أي مستند في ثواني. ده يرفع قيمة الأرشيف الرقمي بشكل كبير جدًا.

**التقدير:** متوسط/عالي الجهد — ثلاثة أسابيع.

---

### 4.3 تقارير PDF رسمية مع ترويسة المعهد

**الفكرة:** توليد تقارير PDF رسمية بتنسيق احترافي كامل — فيه شعار المعهد والتاريخ والتوقيع الرقمي — جاهزة للطباعة والتقديم الرسمي.

**التطبيق التقني:** Puppeteer أو React-PDF مع template مخصص لمعهد دلتا. التقارير بتتولّد server-side مما بيضمن جودة وتناسق.

**القيمة البيزنس:** التقارير الحالية هي screenshot أساسًا. التقارير الرسمية بتتقدم لجهات رسمية وبتعكس احترافية المعهد. خصوصًا مهم للتقارير الدورية والمراجعات.

**التقدير:** منخفض/متوسط الجهد — أسبوع ونص.

---

### 4.4 تطبيق الجوال (Mobile App)

**الفكرة:** تحويل النظام لتطبيق موبايل يشتغل على Android وiOS بدون الحاجة لإعادة كتابة أي كود.

**التطبيق التقني:** Capacitor.js — بيلفّ التطبيق الويب الحالي في shell أصلي للموبايل. بيضيف Push Notifications ودعم العمل Offline.

**القيمة البيزنس:** الموظفين والمشرفين يقدروا يتابعوا المراسلات ويتخذوا إجراءات من موبايلاتهم في أي مكان. ده بيرفع الكفاءة بشكل ملموس، خصوصًا للإدارة العليا.

**التقدير:** متوسط الجهد — ثلاثة أسابيع مع Testing.

---

### 4.5 سجل تدقيق متقدم (Advanced Audit Log)

**الفكرة:** جدول مستقل بيسجّل كل حاجة بتحصل في النظام — من دخول وخروج المستخدمين، لكل تعديل على أي بيانات، لكل export اتعمل.

**التطبيق التقني:** PostgreSQL Triggers بتكتب في جدول `audit_log` منفصل — تلقائي ومش بيتأثر بالكود.

**القيمة البيزنس:** في حالة أي خلاف أو تحقيق، فيه سجل لا يُنكر لكل حاجة حصلت. ده ضروري جدًا للمؤسسات اللي خاضعة لرقابة أو لوائح.

**التقدير:** منخفض الجهد — أسبوع.

---

### 4.6 لوحة تحكم ذكية مع تحليلات زمنية (Analytics 2.0)

**الفكرة:** ترقية التقارير الحالية بإضافة:
- **فلاتر زمنية:** أسبوع / شهر / ربع سنة / سنة
- **مقارنة الفترات:** الشهر ده مقابل الشهر اللي فات
- **Heatmap:** بيوضح أنهي أيام ووقت في اليوم فيه أكتر نشاط للمراسلات
- **KPI Cards** مع مؤشرات الاتجاه (اتزاد/اتقل)

**التطبيق التقني:** إضافة endpoints جديدة للـ API مع queries معقدة + توسيع مكتبة Recharts الموجودة.

**القيمة البيزنس:** القيادة تقدر تاخد قرارات مبنية على بيانات حقيقية بدل الحدس — فين الـ bottlenecks؟ أنهي قسم عنده تأخيرات؟ أنهي فترة في الشهر فيها ضغط؟

**التقدير:** متوسط الجهد — ثلاثة أسابيع.
