## 📖 القسم الأول: السكريبت العام (لكل الفريق)

### اقرأ ده كويس — هو اللي هيخليك فاهم المشروع كله

---

### المشكلة اللي حليناها

في أي مؤسسة تعليمية كبيرة زي معهد دلتا العالي، في مراسلات رسمية كتير بتحصل كل يوم بين الأقسام المختلفة — طلبات، تقارير، تعميمات، موافقات. المشكلة إن الطريقة التقليدية (ورق أو إيميل عادي) فيها عيوب كبيرة:

- **مفيش تتبع:** مش عارف المراسلة وصلت لمين وعملت إيه فيها
- **مفيش صلاحيات:** أي حد يقدر يشوف أي حاجة
- **مفيش أرشيف منظم:** الأوراق بتضيع
- **مفيش إشعارات:** لازم تتصل أو تمشي بنفسك تسأل

**الحل اللي عملناه:** نظام ويب متكامل بيتحكم في كل دورة حياة المراسلة من لحظة إنشائها للموافقة أو الرفض وحتى الأرشفة النهائية.

---

### الفيتشرز الأساسية للنظام

**1. نظام صلاحيات ثلاثي الطبقات:**
- `admin` (مدير) — عنده كل الصلاحيات
- `supervisor` (مشرف) — يوافق ويرفض ويأرشف
- `employee` (موظف) — يبعت ويشوف مراسلاته بس

**2. صندوق الوارد (Inbox):**
جدول بكل المراسلات مع فلترة بالحالة والنوع وبحث نصي وتصدير Excel.

**3. دورة حياة المراسلة:**
إنشاء ← معلقة ← موافقة/رفض/أرشفة — وكل خطوة بتتسجل في سجل التاريخ.

**4. الأرشيف:**
كل مراسلة اتأرشفت بتتسجل مع سبب الأرشفة ومين أرشفها وإمتى.

**5. لوحة التحكم والتقارير:**
رسوم بيانية بتوضح توزيع المراسلات على الأقسام والحالات.

**6. الإشعارات:**
كل موظف بياخد إشعار لما يجيله مراسلة أو يحصل أكشن على مراسلته.

**7. Super Admin:**
حساب `master@delta.edu.eg` مخفي من قائمة الموظفين، عنده صلاحية إعادة تعيين كلمة مرور أي موظف.

**8. رفع الملفات:**
كل مراسلة ممكن يكون فيها ملف مرفق (PDF أو صورة) بيتخزن على Supabase Storage.

### التقنيات المستخدمة (Tech Stack) — ملخص سريع

| الطبقة | التقنية |
|--------|---------|
| لغة البرمجة | TypeScript (للـ Frontend والـ Backend مع بعض) |
| الواجهة | React 19 + Vite + TailwindCSS + Radix UI |
| الـ Routing | Wouter |
| إدارة البيانات | TanStack React Query |

| التحقق من البيانات | Zod Schemas |
| قاعدة البيانات | PostgreSQL (عبر Supabase) |
| ORM | Drizzle ORM |
| المصادقة | Supabase Auth |
| التخزين | Supabase Storage |
| النشر | VPS + PM2 + Nginx |

## 🎤 الطالب رقم 1 — مقدمة المشروع والمشكلة

**دوره:** يفتح العرض بقوة ويحكي المشكلة اللي جينا نحلها

---

### السكريبت:

> "حضرتك الدكتور الكريم وأعضاء لجنة التحكيم، يسعدنا نقدم لحضراتكم مشروع تخرجنا: **نظام إدارة المراسلات الداخلية** لمعهد دلتا العالي للحاسبات وتكنولوجيا المعلومات.
>
> المشكلة اللي لاحظناها إن المعاهد والمؤسسات التعليمية الكبيرة، زي معهدنا، بيكون فيها تبادل يومي كثيف للمراسلات الرسمية بين أقسامها المختلفة — طلبات إجازات، تقارير دورية، تعميمات إدارية، طلبات موافقات مالية. والمشكلة الكبيرة إن ده كان بيحصل إما على ورق أو على إيميلات عادية مش منظمة، وده بيخلي المؤسسة تعاني من: ضياع المراسلات، غياب التتبع، انعدام الصلاحيات، وبطء في اتخاذ القرارات.
>
> جينا نقدم حل تكنولوجي متكامل: نظام ويب بيرقمن كل دورة حياة المراسلة من لحظة إنشائها، مرورًا بالمراجعة والموافقة أو الرفض، وصولًا للأرشفة النهائية — وكل ده بنظام صلاحيات محكم وواجهة سهلة الاستخدام.
>
> وهسلّم للزميل رقم 2 يشرح تحليل المتطلبات وأنواع المستخدمين اللي صمّمنا النظام حواليهم."

---

### أسئلة الدكتور المتوقعة:

**س1: إيه الفرق بين مشروعكم وبرامج إدارة المراسلات الموجودة تجارياً؟**
> "مشروعنا مصمم خصيصاً لبيئة المعهد — مش نظام جنريك. بيعتمد على هيكل الأقسام الفعلي للمعهد، وبيدعم اللغة العربية كاملاً، ومفتوح المصدر وممكن يتطور مع الوقت حسب احتياجات المؤسسة. فضلاً عن إنه خفيف ومش محتاج لايسنس مدفوع."

**س2: إيه الـ Scope بتاع المشروع؟ يعني إيه اللي عملتوه وإيه اللي مش موجود؟**
> "المشروع بيغطي المراسلات الداخلية بين موظفي المعهد والأقسام. الـ Scope مش بيشمل الربط بالأنظمة الخارجية زي البريد الإلكتروني العادي أو الأنظمة الحكومية — ده ممكن يبقى امتداد مستقبلي للمشروع."

**س3: إيه فائدة الرقم المرجعي في كل مراسلة؟**
> "الرقم المرجعي زي رقم التتبع في الشحن — بيوفر طريقة موحدة ومرتبة للبحث عن أي مراسلة بسرعة بدل ما تدور في الآلاف من السجلات. صيغته `CORR-YYYY-XXXX` وبيتولد تلقائياً من النظام."

## 🎤 الطالب رقم 2 — تحليل المتطلبات وأنواع المستخدمين

**دوره:** يشرح المستخدمين والصلاحيات والـ Use Cases

---

### السكريبت:

> "شكراً للزميل رقم 1. أنا هتكلم عن تحليل المتطلبات.
>
> أول حاجة عملناها في التحليل إننا حددنا **مين هيستخدم النظام** وإيه اللي كل واحد محتاجه. لاقينا إن فيه 3 أنواع مستخدمين أساسيين:
>
> **الأول: الموظف العادي** — محتاج يبعت مراسلات، يشوف الردود عليها، ويستقبل إشعارات. مش محتاج يشوف بيانات كل الناس.
>
> **الثاني: المشرف** — فوق الموظف بدرجة. عنده صلاحية يوافق أو يرفض أو يأرشف المراسلات الواردة ليه ولقسمه، وبيشوف التقارير.
>
> **الثالث: المدير (Admin)** — عنده الكنترول الكامل: إدارة الموظفين، إدارة الأقسام، حذف المراسلات، وعرض كل البيانات.
>
> والنوع الرابع الخاص هو **Super Admin** اللي عنده صلاحيات إدارية فوق إدارية — هيتكلم عنه الزميل رقم 15 بالتفصيل.
>
> هسلّم للزميل رقم 3 يشرح هيكل المشروع التقني الكلي."

---

### أسئلة الدكتور المتوقعة:

**س1: إزاي عملتوا الـ Role-Based Access Control؟ في Frontend ولا Backend؟**
> "عملناه في الاتنين مع بعض. في الـ Frontend، عندنا `ProtectedRoute` بيتحقق من دور المستخدم قبل ما يعرض أي صفحة — لو الدور مش مناسب بيعمله Redirect. وفي الـ Backend، كل Endpoint بيتحقق من الصلاحيات قبل ما يرجع البيانات. يعني الحماية مش في طبقة واحدة بس."

**س2: إيه الفرق عملياً بين الـ Supervisor والـ Admin؟**
> "المشرف بيشوف مراسلات قسمه ويتخذ قرارات على المراسلات الواردة ليه. المدير عنده view كامل على كل الأقسام وبيقدر يضيف ويعدل ويحذف موظفين وأقسام. الفرق الجوهري إن المدير هو من يُنشئ ويُدير البنية الأساسية للنظام."

**س3: لو موظف حاول يدخل على صفحة مش في صلاحيته عن طريق الـ URL مباشرة؟**
> "النظام محمي. `ProtectedRoute` في الـ Frontend بيتحقق من الـ role قبل عرض أي صفحة ولو مش متاحة بيعمل Redirect لصفحة الـ 404 أو الداشبورد. وحتى لو تحايل وعمل Request مباشر للـ API، الـ Backend بيرفض الطلب لأن الـ token مش بيحمل الصلاحية المطلوبة."

## 🎤 الطالب رقم 3 — هيكل المشروع (Architecture & Monorepo)

**دوره:** يشرح هيكل الـ Monorepo وإزاي الأجزاء بتتكلم مع بعض

---

### السكريبت:

> "شكراً. أنا هتكلم عن البنية التقنية الكلية للمشروع.
>
> قررنا نبني المشروع كـ **Monorepo** باستخدام `pnpm workspaces`. يعني إيه ده؟ يعني عندنا أكتر من تطبيق في ريبو واحد بيشاركوا كود مشترك بدل ما يتكلوا على بعض عن طريق نسخ الكود أو APIs إضافية.
>
> المشروع متقسم لـ 4 طبقات: **artifacts** — وفيها الـ Frontend والـ Backend. **lib** — وفيها المكتبات المشتركة زي `db` لتعريف قاعدة البيانات، `api-zod` للـ validation schemas، و`api-client-react` اللي فيها الـ hooks الجاهزة للتعامل مع الـ API.
>
> الفايدة الكبيرة إن تعريف نوع بيانات المراسلة — مثلاً — بنكتبه مرة واحدة في `api-zod`، الـ Frontend والـ Backend بيستخدموه مع بعض، وده بيلغي أي تعارض في البيانات بين الطرفين. لو حاجة اتغيرت في الـ Schema، TypeScript بيقولك فوراً في كل الأماكن اللي محتاجة تتغير.
>
> هسلّم للزميل رقم 4 يشرح تصميم الواجهة."

---

### أسئلة الدكتور المتوقعة:

**س1: إيه الفرق بين Monorepo وMultirepo؟ وليه اخترتوا Monorepo؟**
> "في Multirepo، كل تطبيق في ريبو لوحده وبيتعامل مع التاني كـ external dependency. في Monorepo، كلهم في مكان واحد وبيشاركوا الكود مباشرة. اخترنا Monorepo عشان نتجنب مشكلة نسخ أنواع البيانات بين الـ Frontend والـ Backend، وعشان الـ CI/CD أبسط لما كل حاجة في ريبو واحد."

**س2: إيه دور `api-zod` بالضبط؟**
> "`api-zod` بيعرّف شكل كل request وresponse في النظام باستخدام مكتبة Zod. مثلاً، بيعرّف إن إنشاء مراسلة جديدة لازم يكون فيها subject وtype وsenderId. لو أي طلب وصلل الـ Backend ناقصه بيانات أو فيه نوع بيانات غلط، بيتم رفضه تلقائياً قبل ما يوصل لقاعدة البيانات خالص."

**س3: إزاي نظّمتوا الشغل على Monorepo مع إن الفريق 16 طالب؟**
> "استخدمنا Git Branches — كل feature بتتعمل في branch منفصل ثم Pull Request. pnpm بيتأكد إن كل developer عنده نفس إصدارات الـ packages عبر `pnpm-lock.yaml`. والـ TypeScript بيكتشف أي conflict في الأنواع في وقت الكتابة قبل ما يحصل merge."

## 🎤 الطالب رقم 4 — تصميم الواجهة (UI/UX)

**دوره:** يشرح الـ Design System والتقنيات المرئية

---

### السكريبت:

> "شكراً. أنا مسؤول عن جزء تصميم الواجهة.
>
> في تصميم أي نظام مؤسسي، الـ UI لازم يحقق معادلة صعبة: **احترافي بس مش معقد، بسيط بس مش فقير**. استخدمنا **TailwindCSS** للـ Styling، وده خلانا نكتب الـ CSS مباشرة في الـ Components بدل ملفات CSS منفصلة، وده وفّر علينا وقت كتير في الاتساق.
>
> للكومبوننتس الجاهزة استخدمنا **Radix UI** — ده بيوفرلنا عناصر زي الـ Dialogs والـ Dropdowns والـ Select Menus وهي Accessible تلقائياً يعني بتشتغل صح حتى مع قارئات الشاشة. بنركز في التصميم على الألوان الدلالية: الأخضر للموافقة، الأحمر للرفض، الأصفر للتعليق.
>
> دعمنا كمان الـ RTL نظراً لإن النظام كله عربي — كل الـ layout اتصُمم بالاتجاه الصح من اليمين لليسار.
>
> هسلّم للزميل رقم 5 يشرح صفحة تسجيل الدخول."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه استخدمتوا Radix UI بدل تعملوا الكومبوننتس بنفسكم؟**
> "بناء كومبوننتس زي الـ Dialog والـ Select من الصفر بشكل صحيح ومتوافق مع معايير الـ Accessibility بياخد وقت طويل. Radix UI بيحل المشكلة دي ويوفر عناصر محكمة ومختبرة، وإحنا بنركز طاقتنا على الـ Business Logic مش على إعادة اختراع العجلة."

**س2: إيه مكتبة الأيقونات اللي استخدمتوها وليه؟**
> "استخدمنا **Lucide React** — مكتبة أيقونات SVG بتيجي كـ React Components مباشرة. ميزتها إنها خفيفة جداً، بتدعم الـ Tree Shaking يعني بس الأيقونات اللي بتستخدمها بتتبعتلك في الـ Build، وبتقدر تتحكم في حجمها ولونها بسهولة كبيرة."

**س3: إزاي عملتوا الـ RTL Support؟**
> "TailwindCSS بيدعم الـ RTL بشكل تلقائي لما تحط `dir='rtl'` على الـ HTML element. ده بيعكس كل الـ margins والـ paddings والـ flexbox directions تلقائياً. فضلاً عن إننا اخترنا Arabic localization في كل النصوص من البداية."

## 🎤 الطالب رقم 5 — تسجيل الدخول والمصادقة (Auth)

**دوره:** يشرح صفحة Login وسياق المصادقة (AuthContext)

---

### السكريبت:

> "شكراً. أنا هشرح إزاي بيحصل تسجيل الدخول في النظام.
>
> صفحة الـ Login بسيطة: إيميل وباسوورد. لما المستخدم بيضغط دخول، النظام بيبعت البيانات لـ **Supabase Auth** — اللي هو خدمة المصادقة. لو البيانات صح، Supabase بيرجع **JWT Token** — وده الـ Token ده اللي بيثبت هوية المستخدم في كل الـ Requests الجاية.
>
> بعد الـ Token، النظام بيجيب **بروفايل المستخدم** من جدول `employees` في قاعدة البيانات — الاسم، الدور، القسم. البيانات دي بتتخزن في **AuthContext** — وده Context بيبقى متاح في كل الواجهة. أي صفحة محتاج تعرف مين الفيوزر الحالي بتقول `useAuth()` وبتاخد البيانات.
>
> حمايناه برضو من مشكلة الـ Race Conditions: لو فيه أكتر من event جاي في نفس الوقت، الكود بيتأكد إن الـ profile fetch مش بيحصل مرتين في نفس اللحظة، وفيه Timeout بـ 6 ثواني عشان لو قاعدة البيانات بطيئة مش يفضل النظام معلق للأبد.
>
> هسلّم للزميل رقم 6 يشرح الداشبورد."

---

### أسئلة الدكتور المتوقعة:

**س1: إيه هو الـ JWT Token وإزاي بتستخدموه؟**
> "JWT اختصار لـ JSON Web Token — ده String مشفر بيحتوي على بيانات المستخدم (زي الـ ID والـ Email) وتاريخ انتهاء الصلاحية. بيتبعت في كل Request في الـ Header. السيرفر بيفك تشفيره ويتأكد منه من غير ما يرجع يسأل قاعدة البيانات في كل مرة. ده بيخلي الـ Authentication سريع وقابل للـ Scale."

**س2: إيه هو الـ AuthContext وليه عملتوه؟**
> "الـ AuthContext هو مخزن مركزي لبيانات المستخدم الحالي متاح في كل الـ App. بدلاً من إن كل Component يعمل fetch منفصل لبيانات المستخدم، بنعمل fetch مرة واحدة عند تسجيل الدخول ونخزن النتيجة في الـ Context. أي Component محتاجها بيستخدم `useAuth()` hook ويجيب البيانات فوراً من الذاكرة."

**س3: إيه اللي بيحصل لو الـ Token اتسرق حد تاني؟**
> "ده سيناريو الـ Token Theft. حمايتنا الأولى إن Supabase بيخزن الـ Token في الـ httpOnly Cookie لو استخدمنا الـ SSR — أما في الـ SPA زي مشروعنا بيتخزن في الـ Memory أو الـ LocalStorage. الحماية الأقوى إن كل Token عنده Expiry Time قصير وبيتجدد تلقائياً. لو سيناريو الاختراق حصل، المدير يقدر يعمل Sign Out Everywhere من Supabase Dashboard."

## 🎤 الطالب رقم 6 — لوحة التحكم والإحصائيات (Dashboard)

**دوره:** يشرح صفحة الداشبورد والرسوم البيانية

---

### السكريبت:

> "شكراً. أنا هتكلم عن صفحة لوحة التحكم — وهي أول صفحة بتشوفها بعد تسجيل الدخول.
>
> الداشبورد مش مجرد ديكور — هو **مركز القرار**. بيديك نظرة سريعة على حالة المؤسسة من مراسلات. فيه **4 بطاقات إحصائية** في الأعلى بيتعرض فيها: إجمالي المراسلات، المعلقة، الموافق عليها، والمؤرشفة.
>
> تحتهم **رسم بياني دائري** بيوضح نسبة كل حالة من الحالات، و**رسم بياني عمودي** بيوضح توزيع المراسلات على الأقسام — وده مهم جداً لإن المدير يقدر يشوف أنهي قسم عنده load أكتر.
>
> في الأسفل، **جدول المراسلات الأخيرة** بيعرض أحدث 10 مراسلات مع روابط مباشرة لتفاصيلها.
>
> الرسوم البيانية مبنية بمكتبة **Recharts** والبيانات بتيجي من API endpoints مخصصة في الـ Backend لكل widget.
>
> هسلّم للزميل رقم 7 يشرح صندوق الوارد."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه استخدمتوا Recharts بالتحديد؟**
> "Recharts بتيجي كـ React components مباشرة، يعني بتقدر تدي الـ Chart بيانات كـ props وتظهر تلقائي. مش محتاج تتعلم API معقدة زي D3.js. بتدعم الـ Responsive Design تلقائياً وبتوفر كل أنواع الـ Charts اللي احتجناها: Pie Chart، Bar Chart، وLine Chart."

**س2: الداشبورد بيعرض نفس البيانات لكل المستخدمين؟**
> "لأ. البيانات بتتفلتر حسب دور المستخدم. الموظف العادي بيشوف بس مراسلاته هو. المشرف بيشوف مراسلات قسمه. المدير بيشوف الكل. ده بيتحكم فيه الـ Backend اللي بياخد الـ viewerId والـ viewerRole في الـ query parameters ويفلتر بناءً عليهم."

**س3: البيانات بتتحدث تلقائياً؟**
> "البيانات بتتحدث في كل مرة المستخدم يفتح الصفحة. استخدمنا React Query اللي بيعمل background refetch بشكل ذكي. لو عايزين real-time updates، نقدر نضيف Supabase Realtime Subscriptions كـ امتداد مستقبلي للمشروع."

## 🎤 الطالب رقم 7 — صندوق الوارد وإضافة مراسلة

## 🎤 الطالب رقم 8 — تفاصيل المراسلة وسجل التاريخ

**دوره:** يشرح صفحة CorrespondenceDetail وسجل الأكشنز

---

### السكريبت:

> "شكراً. هتكلم عن صفحة تفاصيل المراسلة — وهي الصفحة اللي بتدخل عليها لما بتضغط على أي مراسلة.
>
> الصفحة دي بتعرض **كل تفاصيل المراسلة المحددة**: الموضوع، النص الكامل، المرسل، المستلم، التاريخ، الحالة، الأولوية، والملف المرفق لو موجود.
>
> الجزء المهم هو **سجل التاريخ (History Log)** — بيوضح كل أكشن حصل على المراسلة دي بالترتيب الزمني: مين عمله، إمتى، وإيه الملاحظات. ده بيوفر شفافية كاملة ومساءلة.
>
> **أزرار الأكشن** بتظهر أو تختفي حسب دور المستخدم وحالة المراسلة الحالية. المشرف والمدير بيشوفوا أزرار الموافقة والرفض والأرشفة. المدير بس بيشوف زر الحذف. الموظف العادي بيشوف المراسلة فقط بدون أزرار تعديل.

> هسلّم للزميل رقم 9 يشرح الأرشيف."

---

### أسئلة الدكتور المتوقعة:

**س1: لماذا من المهم تسجيل تاريخ كل أكشن؟**
> "سجل التاريخ هو **Audit Trail** — بيضمن المساءلة. لو حد ادعى إن مراسلة اتوافق عليها أو اترفضت بغلط، النظام عنده دليل دقيق: مين عمل الأكشن ده، إمتى بالظبط، وإيه الملاحظات اللي كتبها. ده مهم جداً في البيئات الإدارية الرسمية."

**س2: إيه الفرق بين الرفض والأرشفة؟**
> "الرفض يعني إن المراسلة مش موافق عليها لأسباب محددة — وبتفضل ظاهرة في صندوق الوارد بحالة 'مرفوضة' عشان المرسل يشوف السبب ويتصرف. الأرشفة تعني إن المراسلة خلصت دورة حياتها وبنقلها لمكان تخزين منفصل للرجوع ليها في المستقبل بدون ما تزيط في القائمة الرئيسية."

**س3: إزاي بتوصل لصفحة التفاصيل؟**
> "كل مراسلة في الجدول، رقمها المرجعي بيكون رابط `<Link href='/correspondences/{id}'>`. لما بتضغط عليه، Wouter Router بيأخد الـ `id` من الـ URL ويبعت request للـ API `GET /correspondences/:id` وبيرجع التفاصيل الكاملة بما فيها الـ history records."

## 🎤 الطالب رقم 9 — الأرشيف

**دوره:** يشرح صفحة Archive وكيفية عمل الأرشفة

---

### السكريبت:

> "شكراً. أنا هتكلم عن نظام الأرشيف.
>
> الأرشيف مش مجرد سلة مهملات — هو **مستودع رسمي** للمراسلات المكتملة. لما مراسلة تتأرشف، بتتنقل لجدول منفصل اسمه `archive` في قاعدة البيانات. الجدول ده بيحفظ: المراسلة نفسها، مين أرشفها، إمتى، وسبب الأرشفة.
>
> صفحة الأرشيف بتعرض **قائمة بكل المراسلات المؤرشفة** مع إمكانية البحث والفلترة فيها. كل صف ممكن تضغط عليه تشوف تفاصيل المراسلة الكاملة.
>
> الصفحة دي متاحة للمدير والمشرف بس — الموظف العادي مش محتاج يشوف الأرشيف.
>
> فيه كمان **Cascade Delete** في قاعدة البيانات: لو المراسلة الأصلية اتحذفت، سجلات الأرشيف المرتبطة بيها بتتحذف تلقائياً من غير ما نحتاج نعمل حاجة إضافية.
>
> هسلّم للزميل رقم 10 يشرح صفحة التقارير والتصدير."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه عملتوا جدول archive منفصل بدل ما تحطوا flag في جدول المراسلات؟**
> "الجدول المنفصل بيوفرلنا معلومات إضافية مرتبطة بعملية الأرشفة نفسها (مين أرشف وإمتى وليه) من غير ما نضخم جدول المراسلات بأعمدة اختيارية. ده بيحسن الـ query performance لإن queries الـ inbox الرئيسية مش بتشوف البيانات المؤرشفة، وبيفصل المسؤوليات بشكل أنظف."

**س2: هل ممكن تسترجع مراسلة من الأرشيف؟**
> "في النسخة الحالية، الأرشفة نهائية — وده تصميم مقصود لأن الأرشيف يُفترض أن يكون سجلاً غير قابل للتعديل. لكن ممكن كامتداد مستقبلي نضيف زر 'استرجاع' للمدير بيغير حالة المراسلة لـ 'active' تاني."

**س3: الأرشيف بيأثر على الـ Performance لو البيانات كتير؟**
> "قاعدة البيانات عندها Indexes على `correspondenceId` وعلى `createdAt`. وبما إن الأرشيف جدول منفصل، queries الـ Inbox الرئيسية مش بتمسه خالص. وده بيعني إنه حتى لو الأرشيف فيه ملايين السجلات، مش هيبطّء صندوق الوارد."

## 🎤 الطالب رقم 11 — إدارة الأقسام والموظفين

**دوره:** يشرح صفحتي Departments وEmployees

### السكريبت:

> "شكراً. هتكلم عن إدارة الأقسام والموظفين — وهما الصفحتين اللي بيبني عليهم كل النظام.

> **صفحة الأقسام:** بتمكّن المدير يضيف أقسام جديدة للمعهد بـ: اسم القسم، كوده المختصر، ولون مميز. الألوان بتستخدم في كل النظام لتمييز المراسلات بصرياً. ممكن كمان يعدل أو يحذف أقسام.

> لما موظف بيتحذف، المراسلات المرتبطة بيه مش بتتحذف — بس الـ sender أو receiver بيبقى `NULL` في قاعدة البيانات، وده بيضمن إن السجل التاريخي محفوظ.

> "أول حاجة النظام بيبعت request للـ Backend. الـ Backend بيستخدم Supabase Admin Client يعمل حساب جديد في Supabase Auth (إيميل + باسوورد) بـ `email_confirm: true` عشان المستخدم يقدر يدخل فوراً. بعد ما الـ Auth Account اتعمل، بيضيف صف جديد في جدول `employees` في قاعدة البيانات بنفس البيانات. ده بيخلي الـ Auth والـ DB دايماً متزامنين."

## 🎤 الطالب رقم 12 — الباك إند ومسارات الـ API

**دوره:** يشرح بنية Express.js وكل الـ Routes

---

### السكريبت:

> "شكراً. هتكلم عن الباك إند — قلب النظام اللي مش بيظهر للمستخدم بس كل حاجة بتمر عليه.
>
> الباك إند مبني بـ **Express.js** على **Node.js** مكتوب كله بـ TypeScript. هيكله بسيط ومنظم: كل Resource عنده ملف Routes منفصل.
>
> عندنا 7 ملفات Routes: **employees** لإدارة الموظفين، **departments** للأقسام، **correspondences** للمراسلات والأكشنز عليها، **archive** للأرشيف، **dashboard** للإحصائيات، **notifications** للإشعارات، و**health** للتأكد إن السيرفر شغال.
>
> كل Endpoint بيعمل 3 حاجات: يتحقق من البيانات الجاية باستخدام Zod، يعمل العملية على قاعدة البيانات عبر Drizzle ORM، ويرجع الـ Response. لو حصل خطأ في أي خطوة، بيرجع error code واضح مش بيكشف تفاصيل حساسة.
>
> السيرفر بعد الـ Build بيتجمع في ملف `dist/index.mjs` واحد وبيشتغل بـ PM2 على السيرفر.
>
> هسلّم للزميل رقم 13 يشرح الأمان والمصادقة."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه استخدمتوا Express 5 وليس Express 4؟**
> "Express 5 بيدعم async/await بشكل native في الـ Route Handlers — يعني لو حصل خطأ في دالة async، Express 5 بيمسك الخطأ ده تلقائياً ويبعته لـ Error Handler من غير ما نحتاج نلف الكود في try-catch في كل route. ده بيخلي الكود أنظف وأقل عرضة للـ Unhandled Promise Rejections."

**س2: إيه هو الـ Middleware وبتستخدموه إزاي في المشروع؟**
> "الـ Middleware هو كود بيشتغل على كل Request قبل ما يوصل لـ Route Handler. في مشروعنا، CORS Middleware بيتحكم مين يقدر يكلم الـ API. Pino Middleware بيسجل كل Request في الـ Log. وممكن نضيف Auth Middleware يتحقق من الـ JWT في كل Request."

**س3: إزاي بتتعاملوا مع الـ Errors في الـ Backend؟**
> "كل Endpoint بيعمل validation أول بأول. لو البيانات الجاية غلط بيرجع status 400 مع وصف الخطأ. لو مش موجود بيرجع 404. لو مش مصرح بيرجع 403. الـ Error messages واضحة بس مش بتكشف تفاصيل الـ Database أو الـ Stack Trace للمستخدم."

## 🎤 الطالب رقم 13 — الأمان والمصادقة (Supabase Auth + Security)

**دوره:** يشرح Supabase Auth وطبقات الأمان

---

### السكريبت:

> "شكراً. هتكلم عن أمان النظام، وده جزء مش اختياري في أي نظام مؤسسي.
>
> استخدمنا **Supabase Auth** كخدمة مصادقة متكاملة. ليه؟ لأنه بيوفرلنا: تشفير آمن لكلمات المرور بـ bcrypt، إدارة الـ JWT Tokens بشكل تلقائي، وإمكانية إدارة المستخدمين من Dashboard مرئي.
>
> **طبقات الأمان في النظام:**
>
> الطبقة الأولى: **Authentication** — مش ممكن أي request يعدي من غير JWT Token صالح.
>
> الطبقة الثانية: **Authorization** — كل endpoint بيتحقق إن المستخدم عنده الصلاحية المناسبة، مش بس إنه logged in.
>
> الطبقة الثالثة: **Input Validation** — كل البيانات الجاية بتتحقق منها بـ Zod قبل ما توصل قاعدة البيانات، وده بيمنع SQL Injection وإرسال بيانات غلطة.
>
> الطبقة الرابعة: **CORS** — السيرفر بيقبل requests من الـ Frontend domain بس.
>
> هسلّم للزميل رقم 14 يشرح قاعدة البيانات."

---

### أسئلة الدكتور المتوقعة:

**س1: إزاي Supabase بيخزن كلمات المرور؟**
> "Supabase بيستخدم `bcrypt` لتشفير كلمات المرور — وده One-Way Hashing يعني مش ممكن تفك التشفير وترجع للباسوورد الأصلي. حتى لو قاعدة البيانات اتسرقت، الهاكر مش هيقدر يعرف كلمات المرور الحقيقية. لو حد نسي باسووردو، الحل الوحيد هو إعادة تعيين باسوورد جديد."

**س2: إزاي بتحموا من SQL Injection؟**
> "أولاً، Drizzle ORM بستخدم Parameterized Queries — يعني البيانات الجاية من المستخدم مش بتتحط مباشرة في جملة SQL خالص، وده بيمنع SQL Injection بشكل كامل. ثانياً، Zod Validation بترفض أي input مش بالشكل الصح قبل ما يوصل للـ ORM أصلاً."

**س3: الـ CORS إيه وليه مهم؟**
> "CORS يعني Cross-Origin Resource Sharing — بيتحكم في مين يقدر يبعت requests للـ API بتاعتنا. من غيره، أي موقع على الإنترنت يقدر يعمل requests للـ API بتاعتنا باستخدام بيانات أي مستخدم logged in. بعد ما ضبطنا CORS، الـ API بيقبل requests من domain معين بس — وده بيمنع هجمات CSRF."

## 🎤 الطالب رقم 14 — قاعدة البيانات والـ Schema (Drizzle ORM)

**دوره:** يشرح الجداول وعلاقاتها وDrizzle ORM

---

### السكريبت:

> "شكراً. هتكلم عن قاعدة البيانات — الأساس اللي النظام كله واقف عليه.
>
> قاعدة البيانات مبنية على **PostgreSQL** بتشتغل عبر Supabase. استخدمنا **Drizzle ORM** عشان نتعامل معاها — وده بيخلينا نكتب TypeScript بدل SQL خالص، والكود بيتحول لـ SQL محسوب في الـ Background.
>
> عندنا **6 جداول**: `employees` للموظفين، `departments` للأقسام، `correspondences` للمراسلات الأساسية، `correspondence_history` لسجل كل أكشن، `archive` لسجلات الأرشفة، و`notifications` للإشعارات.
>
> العلاقات بين الجداول مهمة: `correspondences` بيرتبط بـ `employees` للمرسل والمستلم بـ `onDelete: SET NULL` — يعني لو موظف اتحذف، المراسلة مش بتتحذف. بس `correspondence_history` و`archive` بيرتبطوا بـ `correspondences` بـ `onDelete: CASCADE` — يعني لو المراسلة اتحذفت، سجلاتها بتتحذف معاها تلقائياً.
>
> هسلّم للزميل رقم 15 يشرح ميزة السوبر أدمن والإشعارات."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه استخدمتوا Drizzle ORM وليس Prisma مثلاً؟**
> "Drizzle بيتميز بإنه أخف وزناً وأسرع في الـ Runtime مقارنة بـ Prisma. الـ Schema بيكون TypeScript Code خالص — مش DSL جديد لازم تتعلمه. وبيدعم SQL المعقد بشكل كامل لما محتاجين نكتب queries صعبة. فضلاً عن إن الـ Bundle Size أصغر بكتير وده مهم على السيرفر."

**س2: إيه هو الـ Migration وإزاي بتستخدموه؟**
> "الـ Migration هو ملف SQL بيوصف التعديلات اللي محتاج تعملها في قاعدة البيانات عند الترقية. مثلاً، لو ضفنا عمود جديد، بنعمل migration file بيضيف العمود ده. ده بيضمن إن قاعدة البيانات على أي environment (محلي أو سيرفر) بتتحدث بنفس الطريقة بدل ما حد يعمل تعديل يدوي."

**س3: ليه UUID وليس Integer كـ Primary Key؟**
> "UUID بيوفر مزايا كتير: مش ممكن تخمنه — يعني مش ممكن حد يعمل `/correspondences/1` ويجرب `/correspondences/2` وهكذا. وبيسمح بإنشاء IDs من غير ما نرجع للـ Database لنعرف الرقم الجاي. وبيسهل الـ Merge لو في المستقبل كانت في databases متعددة."

## 🎤 الطالب رقم 16 — النشر على السيرفر (Deployment)

**دوره:** يشرح كيفية النشر على VPS بـ PM2 وNginx

---

### السكريبت:

> "شكراً. وده آخر جزء في العرض — إزاي النظام بيشتغل على سيرفر حقيقي.
>
> الكود مرفوع على **GitHub** واللينك هو `github.com/alrrakb/nabil`. على السيرفر، بنعمل `git clone` لجيب الكود، بنثبت الـ packages بـ `pnpm install`، وبنعمل Build للـ Frontend والـ Backend.
>
> بعد الـ Build، الـ Backend بيبقى ملف `.mjs` واحد — ده مش محتاج Compilation تاني. الـ Frontend بيبقى ملفات HTML وCSS وJS جاهزة.
>
> بنستخدم **PM2** لإدارة العمليات في الخلفية — هو بيضمن إن السيرفر بيشتغل طول الوقت، ولو وقع بيشيل نفسه تلقائياً. وبعد restart السيرفر، PM2 بيرجع يشغل كل حاجة تلقائياً.
>
> بنستخدم **Nginx** كـ Reverse Proxy — بيستقبل الـ Requests على Port 80 ويوزعها: المسارات اللي تبدأ بـ `/api` بتروح للـ Backend على Port 8080، والباقي بيروح للـ Frontend على Port 9000.
>
> وده بيختم عرضنا. شكراً لحضراتكم، ونحن مستعدون لأي أسئلة."

---

### أسئلة الدكتور المتوقعة:

**س1: ليه استخدمتوا PM2 وليس تشغيل Node مباشرة؟**
> "لو شغّلنا Node مباشرة بـ `node index.mjs`، لو السيرفر وقع لأي سبب — زي خطأ في الكود أو انقطاع الكهرباء — التطبيق بيوقف ومش بيرجع تلقائياً. PM2 Process Manager متخصص في إدارة التطبيقات: بيراقب كل process، وبيعيد تشغيله تلقائياً لو وقع، وبيحفظ الـ Logs، وبيتأكد إنه بيشتغل بعد كل restart للسيرفر."

**س2: إيه دور Nginx بالظبط في المشروع؟**
> "Nginx بيشتغل كـ Reverse Proxy — واقف في الأمام بيستقبل كل الـ Traffic. بيوزع: الـ Requests للـ Frontend للـ Static Files على Port 9000، والـ Requests للـ API للـ Backend على Port 8080. وبيوفر حماية إضافية عشان الـ Backend مش مكشوف مباشرة على الإنترنت. لو في المستقبل عايزين نضيف SSL (HTTPS)، Nginx هو المكان الصح نضيفه فيه."

**س3: إزاي بتعملوا Update للنظام بعد ما الكود يتغير؟**
> "بسيطة: بنعمل `git pull origin main` على السيرفر، بنعيد البناء بـ `pnpm build`، وبنعمل `pm2 restart` للتطبيق. كل ده بياخد دقيقتين. في المستقبل ممكن نعمل CI/CD Pipeline على GitHub Actions يعمل ده كله تلقائياً كل ما بيحصل push على الـ main branch."

## ❓ القسم الثالث: الأسئلة العامة للفريق كله

*الأسئلة دي ممكن الدكتور يسألها للفريق كله — أي حد فيكم يقدر يجاوب*

---

### السؤال 1: ليه Supabase وليس Firebase؟

> **الإجابة المثالية:**
>
> "السؤال ده مهم ومش بديهي. الاتنين من Firebase وSupabase بيوفروا Authentication وStorage. الفرق الجوهري إن Firebase بيستخدم **NoSQL (Firestore)** بينما Supabase بيستخدم **PostgreSQL** — قاعدة بيانات علائقية حقيقية.
>
> مشروعنا فيه علاقات معقدة بين الجداول: موظف له مراسلات، مراسلة ليها history وأرشيف. الـ Relational Database بتمثل العلاقات دي بشكل طبيعي وبتسمح بـ Joins قوية وForeign Keys مع Cascade Delete. في NoSQL كنا هنضطر نعمل ده يدوياً في الكود. فضلاً عن إن Supabase Open Source بينما Firebase مقفول تماماً على Google."

---

### السؤال 2: إزاي 16 شخص اشتغلوا على كود واحد من غير فوضى؟

> **الإجابة المثالية:**
>
> "ده تحدي تنظيمي حقيقي. الحل اتكون في ثلاث محاور: أولاً **Git + GitHub** كـ Version Control — كل feature بتتعمل في Branch منفصل وبعدين بيحصل Merge. ثانياً **TypeScript** نفسه بيساعد — لو حد غيّر Interface بيأثر على باقي الكود، المحرر بيحس بذلك فوراً قبل ما يحصل تعارض. ثالثاً **تقسيم واضح للمسؤوليات** — الـ Monorepo structure بيخلي كل مجموعة تشتغل على جزء منفصل (Frontend، Backend، DB، UI) بتأثير محدود على التانيين."

---

### السؤال 3: إيه اللي يحصل لو السيرفر وقع؟

> **الإجابة المثالية:**
>
> "عندنا طبقات حماية: PM2 بيراقب التطبيق وبيعيد تشغيله تلقائياً لو وقع. قاعدة البيانات على Supabase Cloud مش على نفس السيرفر — يعني البيانات آمنة حتى لو السيرفر وقع. لو السيرفر نفسه اتعطل كلياً، الحل هو restore الكود من GitHub وتشغيل build من أول — كل الـ Data في Supabase سليمة. في بيئة Production حقيقية، نقدر نضيف Load Balancer وأكتر من Instance للـ Backend للـ High Availability."

---

### السؤال 4: إزاي بتضمنوا أمان البيانات الحساسة؟

> **الإجابة المثالية:**
>
> "البيانات الحساسة زي الـ Supabase Keys والـ Database URL موجودة في ملف `.env` مش موجود في الـ GitHub (موضوع في `.gitignore`). في الـ Frontend، بنستخدم فقط الـ `anon key` اللي صلاحياتها محدودة على الـ Read للبيانات العامة — الـ Service Role Key الأقوى موجودة بس في الـ Backend. كلمات المرور مشفرة بـ bcrypt. والـ JWT Tokens ليها Expiry وبتتجدد."

---

### السؤال 5: إيه هي أبرز التحديات اللي واجهتوها وإزاي حليتوها؟

> **الإجابة المثالية:**
>
> "واجهنا ثلاث تحديات رئيسية: 
>
> الأول: **مشكلة الـ Race Condition في الـ Auth** — لما المستخدم بيسجل دخول، Supabase بيبعت أكتر من event في نفس الوقت، وده كان بيخلي الـ Profile fetch يحصل مرتين ومرة منهم ترجع null وتمسح البيانات الصح. حليناها بـ Guard Logic في الـ `onAuthStateChange` يتحقق من حالة الـ initialization قبل ما يعمل أي حاجة.
>
> الثاني: **تزامن الـ Types بين Frontend والـ Backend** — حليناها بالـ Monorepo ومكتبة `api-zod` المشتركة.
>
> الثالث: **الـ Build والـ Deployment** — حليناه بـ esbuild اللي بيعمل Bundle للكود كله في ملف واحد وبيشتغل بسرعة جداً."

---

### السؤال 6: هل النظام قابل للتطوير مستقبلاً؟ إيه اللي ممكن تضيفوه؟

> **الإجابة المثالية:**
>
> "تصميم النظام من الأساس بيسمح بالتوسع. أبرز الإضافات الممكنة:
> - **Real-time Notifications** بـ Supabase Realtime Subscriptions — الكود جاهز يستقبل الإضافة دي
> - **CI/CD Pipeline** بـ GitHub Actions للـ Auto Deploy
> - **HTTPS/SSL** عبر Let's Encrypt + Nginx — سهل جداً بعد ما الـ Nginx موجود
> - **إرسال إيميل** عند إنشاء مراسلة عبر Supabase Email أو SendGrid
> - **تطبيق موبايل** بـ React Native اللي ممكن يشارك الـ API بسهولة
> - **تكامل مع أنظمة خارجية** زي منظومة التعليم الحكومية"

### السؤال 7: ليه TypeScript وليس JavaScript؟

> **الإجابة المثالية:**
>
> "TypeScript بيضيف Static Type Checking على JavaScript. يعني لو دالة بتتوقع رقم وبعتتلها نص، الـ Editor بيقولك غلط فوراً قبل ما تشغل الكود. في مشروع بحجم 16 شخص وأكتر من 30 ملف، ده بيمنع فئة كاملة من الأخطاء اللي ممكن تظهر بس في الـ Production. فضلاً عن إنه بيخلي الـ Autocomplete والـ Refactoring أسهل بكتير — لما بتغير اسم دالة، محررك بيقولك في كل الأماكن اللي لازم تتغير."
